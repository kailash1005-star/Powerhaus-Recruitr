Recruitr · Engine documentation

How Recruitr finds candidates, and how it scores them

Two halves of one system. Sourcing turns a job description into a LinkedIn search that returns real people. Matching scores those people against what the job actually requires, and shows its evidence. This document covers the architecture, the workflow, every prompt, and the exact scoring maths.

The big picture

The system's organising idea: the requirement drives everything, and it is computed once. A job description is parsed into a structured Role Spec, and that same object aims the LinkedIn search, gates the spend, and grades the results. Sourcing and matching never form separate opinions about what the job needs.

Two properties are worth calling out, because they are what make the numbers trustworthy:

  • The gate sits before the money. A LinkedIn search returns a title and location for free; deep enrichment costs per profile. Anyone whose title has nothing to do with the role is dropped before we pay for them.
  • The LLM never states a fact. Every score, every gap, every skill credit is computed in code. The model only writes the prose. It cannot contradict the evidence, because it is not the source of it.

The Role Spec spine

role_spec_service.py owns one canonical structured requirement per job, stored as a parsed_jds document, keyed by a hash of the JD text and linked to the job by sourceJobId.

Role Spec — shapeparsed_jds
{
  "sourceJobId": "6a58c9b76753c3ece2f47999",
  "textHash":    "<sha256 of the JD text>",
  "rawText":     "<the job description>",
  "requirements": {
    "title":            "Sachbearbeiter Entgeltabrechnung",
    "mustHaveSkills":   ["SAP HR3", "Entgeltabrechnung", "Arbeitsrecht",
                         "Sozialversicherungsrecht", "Lohnsteuerrecht"],
    "niceToHaveSkills": ["Prozessoptimierungen"],
    "minYears":         3,
    "location":         null,
    "seniority":        "Senior",
    "responsibilities": [...]
  },
  "embedding": { "model": "text-embedding-3-small", "dim": 1536, "vector": [...] }
}

What the hash buys: one LLM parse and one embedding per distinct JD. Re-running a match, or opening the discovery form twice, reuses the parse. Editing the JD changes the hash, so a stale spec is never silently reused.

Part 1 · Candidate sourcing

The problem this half exists to solve: a job posting title is employer language; a LinkedIn headline is how a person describes themselves. They are frequently different words for the same job, and searching the posting title verbatim is the single biggest cause of a search returning nobody.

Assembled by sourcing/brief.py from what the database already knows — the recruiter never retypes it. The must-have skills come from the Role Spec, which is what lets the Strategist aim at the requirement rather than the job title alone.

The Strategist

One reasoning call, no tools, run once when the discovery form opens. It translates the role into titles real people carry, and pre-plans a broadening ladder. Model: openai:gpt-4o. If no LLM key is configured it degrades to a literal title prefill with confidence: 0 — it is an enhancement, never a hard dependency.

System / instructionssourcing/strategist.py
You are a technical sourcing strategist. You turn a job opening
into a LinkedIn people-search that returns REAL candidates.

The single most important thing you understand: a job POSTING title is written in
employer/HR language, but you are searching PROFILE titles, which are written in
the member's own words. They are frequently different, and searching the posting
title verbatim is the #1 cause of a zero-result search.

Examples of the translation you must perform:
  • "SAP Consultant FI" → nobody's headline says that. Real titles: "SAP FICO
    Consultant", "SAP FI Consultant", "SAP Finance Consultant", "SAP FI/CO
    Consultant", "Senior SAP Consultant". FI is almost always paired with CO in
    practice — include the FICO variants.
  • "Java Developer II" → the "II" is an internal grade, never a headline. Use
    "Java Developer", "Software Engineer", "Backend Developer".
  • "Growth Marketing Ninja" → a real title is "Growth Marketing Manager", ...
  • "Cloud Architect (AWS)" → "Cloud Architect", "AWS Architect", ...

Rules for the filters you produce:
1. `currentJobTitles` — 4 to 8 titles people ACTUALLY carry on LinkedIn. Include
   the common abbreviation AND the expanded form when both are in use (SAP FICO /
   SAP Finance). Order strongest-match first. Never include internal grades
   (II, L3, Band 4), employment types (Contract, Permanent), or req codes.
2. `searchQuery` — a SHORT fuzzy keyword phrase (2-4 words), NOT the full title.
3. `locations` — a LinkedIn-recognisable place. Prefer the metro/city the job is
   in; use the country when the role is remote or the city is tiny.
4. `seniorityLevel` / `function` / `yearsOfExperience` — infer from the JD, not
   the title alone. LEAVE A FILTER NULL when the JD doesn't support it — every
   filter you add shrinks the result set, and a wrong filter is far worse than a
   missing one.
5. Skills are NOT a filter on this actor, but `mustHaveSkills` in the brief is
   the EXACT list the candidate will later be scored against — a candidate who
   carries none of them cannot pass, however good the title looks. So the
   must-haves are your primary aiming input, not a footnote: encode them in the
   titles and the query (must-have "S/4HANA" → add "SAP S/4HANA Consultant" as a
   title; must-have "Entgeltabrechnung" → "Entgeltabrechner", "Payroll
   Specialist", "Personalsachbearbeiter Entgeltabrechnung"). Never silently drop
   the signal, and never propose titles whose holders would plainly have none of
   the must-haves.
6. If the recruiter named target companies, put them in `currentCompanies`.
7. LANGUAGE. People describe themselves in the language they work in. If the JD
   or its must-have skills are in a language other than English (German
   "Entgeltabrechnung", French "Chargé de recrutement"), the people you want have
   native-language headlines, and English titles will not find them. Emit titles
   in THAT language — add the English variant only where it is genuinely also in
   use locally (e.g. "Payroll Specialist" is common in Germany; "Labour Law
   Clerk" is not). Set `profileLanguages` to that language when the role is
   plainly local. Domain vocabulary is the strongest language signal — trust it
   over the posting title.

Enum filters MUST use one of these codes (emit the CODE, not the label):
  <generated live from the actor's own ENUM_TABLES>

Also produce:
  • `interpretedRole` — one line naming what this job really is, in plain terms.
  • `titleReasoning` — why the posting title does or doesn't work as a search term.
  • `rationale` — one short entry per non-empty filter, saying why.
  • `broadeningLadder` — 3 fallback attempts, tried in order ONLY if the search
    returns zero. Each step carries a COMPLETE filter set, strictly broader than
    the one before.
  • `confidence` — 0..1. Be honest: a vague one-line JD with no location is 0.3.
  • `warnings` — anything the recruiter should know.

Be decisive and specific. Prefer fewer, higher-signal filters.

titleReasoning  : The job title 'Sachbearbeiter Entgeltabrechnung (SAP HR3)' is

Why this matters

Before the must-haves reached the brief, the same JD produced an English-titled search and found data engineers. The German domain vocabulary in mustHaveSkills is what tells the model the people it wants write Entgeltabrechner, not Payroll Clerk.

Filters and the enum vocabulary

The Strategist emits a SearchFilters object whose field names match the actor's input exactly, so an agent cannot invent a filter the actor would reject. Enum values are validated against the actor's own tables and an unrecognised value is dropped rather than passed through — the actor fails an entire run on a bad enum instead of ignoring it.

FilterTypeNotes

currentJobTitlesstring[]OR-matched. The main aiming device. Capped at 10.
locationsstring[]Must be a place LinkedIn recognises — "Europe" is not.
yearsOfExperienceenum code1=<1yr · 2=1–2 · 3=3–5 · 4=6–10 · 5=10+
seniorityLevelenum code120=Senior, 210=Experienced Manager, …
functionenum code1=Accounting · 10=Finance · 12=Human Resources · …

currentCompaniesstring[]A deliberate poach. Respected exactly.
Important

Skills are not a filter on this actor. The only way a must-have skill can influence the search is by being encoded into the titles or the query. That is why rule 5 of the Strategist prompt exists, and why the must-haves must reach the brief.

The Broadener

Called only when a search returns zero. It sees the full attempt history — every filter set tried and what each returned — and proposes the next, strictly broader attempt. Model: openai:gpt-4o-mini. Capped at SOURCING_MAX_BROADEN_ATTEMPTS = 3.

An agent rather than just walking the pre-planned ladder, because the ladder is written blind. Once you know which search died, the reason matters: a dead title needs different relaxation than a dead location.

System / instructionssourcing/broadener.py
You are a sourcing strategist recovering from a FAILED LinkedIn
people-search: every attempt so far returned ZERO candidates.

Given the job brief and the full history of what was tried, produce the NEXT
search attempt. It must be STRICTLY BROADER than every attempt already made —
repeating or narrowing is a wasted paid API call.

THE ONE RULE YOU MUST NOT BREAK: widening means casting a wider net for THIS job.
It NEVER means searching for a different job. The role's domain vocabulary — the
words that make this role this role ("Entgeltabrechnung", "Payroll", "FICO",
"S/4HANA") — must survive every attempt. Relaxing "Entgeltabrechner" to "SAP
Consultant" or "HR Specialist" is NOT broadening: payroll clerks and SAP
consultants are different professions, and that search returns a pile of people
who cannot do the job. Zero results means "widen the net", never "change the
target". If the only way you can widen further is by abandoning the domain, the
answer is `give_up`, not a search for someone else.

Relax in THIS order — cheapest, most speculative filters FIRST, and touch the
titles LAST:
  1. Enum filters (`seniorityLevel`, `yearsOfExperience`, `function`,
     `companyHeadcount`). ALWAYS start here. They are INFERRED from the JD, not
     stated by the member, and profiles routinely leave the underlying data
     blank — so they silently exclude people who DO qualify. Several set at once
     is the single most common cause of a zero-result search.
  2. `currentCompanies` — restricts to a handful of employers.
  3. `locations` — widen the city to the metro, then to the country.
  4. `profileLanguages` — only if the role plausibly exists in English too.
  5. TITLES, and only once 1-4 are exhausted. Generalise WITHIN the domain and
     keep the domain word: "Sachbearbeiter Entgeltabrechnung" → "Entgeltabrechner"
     → "Payroll Specialist" → "Payroll" (all still payroll). Never climb out to a
     generic title ("Consultant", "Specialist", "Manager") — those match everyone
     and nobody.

Widen ONE dimension per attempt where possible, so the result stays attributable.

Choose `decision: "broaden"` whenever ANY filter is still left to relax.
Choose `decision: "give_up"` ONLY when the last attempt was already a bare title
plus a country with no enum filters and STILL returned zero.

`reasoning` is shown to the recruiter: one plain sentence on what you changed and
why, referencing the actual values. No jargon.
Design note

decision is an action enum (broaden / give_up) rather than a boolean on purpose. A negated flag (shouldRetry: false) gets inverted by the model, which then returns a perfectly good filter set flagged to be thrown away. Naming both outcomes removes the polarity it has to infer.

The domain guard

The prompt above is not trusted on its own. Observed live: told the titles were too specific, the model relaxed a German payroll clerk into SAP Consultant / HR Specialist and returned twelve SAP Finance consultants. So the rule is also enforced in code.

anchor  = non-generic tokens of the FIRST attempt's titles
          (the Strategist's read of the job, before any relaxation)

          "Entgeltabrechner", "Payroll Specialist",
          "Personalsachbearbeiter Entgeltabrechnung", ...
              ↓ strip generic role words
          {entgeltabrechnung, payroll, personalsachbearbeiter, lohnabrechnung, ...}

for each proposed title:
    keep it only if some anchor word (or an obvious inflection) appears in it

  "Payroll Consultant"  → payroll  ✓ KEEP
  "SAP Consultant"      → —        ✗ DROP
  "HR Specialist"       → —        ✗ DROP   ("specialist" is a generic role word)

if nothing survives → fall back to the Strategist's planned ladder

Generic role words — consultant, berater, specialist, manager, sachbearbeiter, analyst, … — are stripped from the anchor because every profession shares them. Without that, Payroll Specialist and HR Specialist look related.

Result

After the guard, the same search relaxed seniorityLevel, function and the location, kept every payroll title, and returned twelve genuine Personalsachbearbeiter Entgeltabrechnung.

The pre-screen gate

A search hit arrives with a title, company and location, for free. Turning it into something scoreable costs an Apify profile scrape each. The gate spends the free signal first.

It is deliberately lopsided. A false drop is unrecoverable — that person is never enriched, never scored, never seen again. A false keep costs one scrape and is caught by the real matcher a minute later. So it only rejects what it is confident about, and answers "is this person plausibly in the right job family?" — never "do they meet the requirements", because a title cannot evidence a skill list.

SignalWeight
Title vs the titles the search originally aimed atStrongest — the Strategist already translated the role
A must-have skill named in the titleFull credit — direct domain evidence
Title vs the role's own titleFallback when the aim is unusable
No title at allkept
No Role Spec to judge againstkept
Critical

The gate screens against the role, never against used_filters. Screening against the Broadener's relaxed titles let broadening silently lower the bar: the drifted SAP-Finance results scored 100/100 and 10 of 12 were kept. Anchored to the role, 2 of 12 survive.

Calibration, on 39 real candidates

Threshold PRESCREEN_MIN_SCORE = 25 was set against the people actually sourced for a live SAP Consultant job — not invented:

TitleScoreVerdict
SAP Inhouse Consultant100keep
Associate Consultant SAP FI/CO100keep
Senior Consultant (no SAP)50keep
CEO · Geschäftsführer · Interim Manager0drop

It keeps every genuine SAP consultant, drops only executive slop, and drops ~93% of those same people when aimed at an unrelated payroll role. Dropped candidates are stored with their verdict and isAccepted: false — visible and recoverable. Only the spend is withheld.

Enrichment

Survivors are deep-scraped via Apify into apifyEnrichment.profile: full name, headline, location, current title/company, totalYears, skills, titles, experience (with per-role skills and dates), education, certifications — plus the LinkedIn profile photo, in four sizes.

Photos

The CDN URLs are public — no cookie or referer needed — but signed and expiring (a few weeks). Every size is signed separately, so a 200px variant cannot be derived from the 800px URL by rewriting the path. The UI falls back to initials when a link dies.

Weights, and why they move

Four components. The nominal weights:

Profile fit
0.50
Must-have skills
0.30
Experience
0.12
Location
0.08

A component the JD never stated is not scored as a free 100 — it is dropped, and its weight is redistributed over the components that do apply. This is the rule that makes scores mean something. A vague JD naming no location used to hand every candidate the same 8 points of padding, so only profile fit could discriminate, and everyone clustered at the same number.

ComponentApplies whenNominalEffective (payroll JD)
Profile fitalways0.500.543
Must-have skillsthe JD lists any0.300.326
ExperienceminYears is set0.120.130
Locationthe JD names one0.080 — excluded

How each component is valued

  • Profile fit — cosine similarity between the JD embedding and the candidate's profile embedding (text-embedding-3-small, 1536 dims). The only component that weighs the whole document rather than a checklist. Shown to recruiters as Strong / Good / Moderate / Weak.
  • Must-have skills — sum of per-skill credit ÷ number of must-haves. See below.
  • Experience — 1.0 when totalYears ≥ minYears; 0.5 when the CV states no total; otherwise years / minYears.
  • Location — 1.0 on a match, 0.6 when the candidate's location is unknown, 0.3 when clearly elsewhere.

Skill matching rules

Every must-have is credited 0–1, with the evidence recorded. Every rule is evaluated and the strongest wins — not first-match-wins, which let a weak rule cap a strong one.

Two guards that took real bugs to find

The “R” bug

The original matcher used a bare substring test in both directions. The one-letter skill "R" is a substring of arbeitsrecht, sap hr3, entgeltabrechnung — every German compound. Every data engineer scored 100% must-have coverage on an SAP payroll role. Fixed by requiring token boundaries.

German compounds

But token boundaries are too strict for German: Lohnsteuer sits inside Lohnsteuerrecht as a prefix, not a token. A minimum-length floor of 5 characters separates the two — long enough that the overlap can't be coincidence, which credits Lohnsteuer while still blocking "R".

token_sort_ratio is used rather than token_set_ratio: the latter returns 100 whenever one token set is a subset of the other, which is exactly the over-crediting the containment rules above handle deliberately.

The evidence pool

Skills are not the only evidence a candidate has a skill. Scoring profile.skills alone reported Entgeltabrechnung as missing for a woman whose job title is Personalsachbearbeiterin Entgeltabrechnung — the requirement is literally her job.

evidence pool = skills                  ← first, so a real skills hit is named as the source
              + currentTitle
              + titles[]
              + experience[].title

Order matters: skills come first so an exact skills match wins the evidence race and is quoted as the source, rather than the title.

The must-have ceiling

Prose cannot buy a candidate past the bar their missing must-haves set. Coverage caps the final score:

CoverageCeiling
100%100 — no cap
≥ 75%85
≥ 50%65
≥ 25%50
> 0%40
0%25

Because a ceiling flattens many candidates onto the same displayed number, ranking tie-breaks on the uncapped base score — the ordering underneath still carries signal.

Gaps vs partials — and what the LLM is allowed to say

A gap is an absence: nothing in the profile evidences it. A partial is real evidence that falls short. Conflating them — treating everything under full credit as a gap — produced a screen that said “Missing specific experience with SAP HR3” directly above evidence crediting SAP HR at 92%.

gaps    = [ credit == 0 ]              →  "Missing"
partial = [ 0 < credit < 1 ]           →  "Partially met"
full    = [ credit >= 1 ]              →  counted, not shown as a problem

invariant: every must-have is exactly one of the three — never both, never neither.
The structural fix

The reasoning model used to be handed the (wrong) gap list as missingMustHave, and its prose then overrode the deterministic data. It wasn't reasoning from the profile — it was rationalising a bad input. Now the scorer is authoritative and the model narrates in reasons only. It cannot restate a fact, so it cannot contradict one.

Bias control

The reasoning prompt is given only skills and experience — never name, contact details or demographics. Candidates are passed anonymised, by id.

Known traps

Live findings, kept here because each one cost real time to diagnose.

Quota looks like “no candidates”

The Apify actor reports an account refusal as Status: SUCCEEDED, Message: free user run limit reached with an empty dataset. Read naively, that is indistinguishable from “nobody matches your search” — so the recruiter is told their role has no candidates, and the Broadener burns its entire budget widening searches that never ran. _raise_if_quota_exhausted now inspects statusMessage and raises instead.

Rule: before blaming filters or titles for a zero-result search, replay a known-good filter set and prove the actor still returns anything. A bare {searchQuery: "Payroll", locations: ["Germany"]} returning 0 is never a real result.
